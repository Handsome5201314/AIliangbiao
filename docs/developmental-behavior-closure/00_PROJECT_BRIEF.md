# 00 项目目标与边界

## 项目定位

本项目基于现有 `Handsome5201314/AIliangbiao` 仓库，分阶段建设“儿童发育行为评估 -> 健康教育 -> 随访复测 -> 医生复核 -> 科研导出 -> Skill/MCP 调用”的闭环系统。

它不是普通问卷系统，也不是普通 AI 聊天机器人。正确方向是：医生端门诊评估为主，家长端居家复测为辅，科研端沉淀历史导入和研究衍生表，比赛端通过 Skill + MCP 展示外部智能体调用链。四类入口共用同一套量表目录、评估会话、确定性计分、医生复核、正式报告、随访和科研数据底座。

## 目标用户

- 医生/科室团队：创建儿童档案、发起门诊评估、查看结果、完成复核、生成报告、管理随访。
- 家长/照护者：通过医生邀请链接或账号登录完成量表，查看医生复核后的报告和人工审核健康教育内容。
- 科研人员：导入历史对照组，导出脱敏研究衍生表，分析 1 个月/3 个月复测和干预效果。
- 外部智能体/比赛平台：通过 Skill + MCP 查询量表、创建会话、提交答案、获取确定性计分和演示解释。

## 首批量表

第一批只聚焦：

- `M_CHAT_R`
- `ABC`
- `CARS`
- `SRS`
- `SNAP-IV`
- `ATEC`
- `VINELAND_3`

当前仓库中七个目标量表均已在 `lib/schemas/core/registry.ts` 注册，并通过 `lib/scales/catalog.ts` 暴露为儿童临床量表。量表授权状态尚未落库，商业化前必须通过 `license_status` 或等价字段完成授权核验。

## 成功标准

- 不重写技术栈，继续使用 Next.js + TypeScript + Prisma + PostgreSQL。
- 医生端、家长 H5、科研端、Skill/MCP 四类入口共享同一套业务数据流。
- 量表最终计分和风险等级由确定性代码或 MCP API 完成，LLM 不参与最终判定。
- 正式报告必须医生复核后才可见，医生复核不能被跳过。
- 科研导出默认脱敏，不输出姓名、电话、身份证号、门诊号、住院号等直接身份字段。
- 后续每个 Phase 都有明确验收标准、测试建议和回滚边界。

## 明确不做

Phase 0 不实现大规模业务功能，不修改 Prisma schema，不新增外部服务依赖，不接入真实短信，不生成正式医学诊断，不把未授权量表写成“已授权”，不删除现有核心功能。
