# 移动端 H5 登录与用户隔离 — 开发提示词

> 本文档为 mobile-h5-prototype 补充登录认证与用户隔离能力的开发指引，基于现有后端认证体系（AUS Token + SMS 验证码）和三种真实使用场景设计。

---

## 一、背景与现状

### 1.1 当前原型问题

mobile-h5-prototype 当前没有任何认证逻辑：App.tsx 无用户状态、无登录页面、无路由守卫、无 Token 管理。家长端和医生端通过顶部 pill 切换，任何人均可自由切换角色，数据完全无隔离。初始化时一次性加载家长和医生双侧数据，service 层使用 `session-mock` 硬编码。

### 1.2 已有后端能力

| 能力 | 状态 | 关键文件 |
|------|------|----------|
| AUS Token 签发/验证 | ✅ 已实现 | `lib/auth/app-session.ts` — HMAC-SHA256, 7天TTL |
| SMS 验证码登录 | ✅ 已实现 | `app/api/auth/login-sms/route.ts` — 腾讯云SMS |
| 密码登录 | ✅ 已实现 | `app/api/auth/login/route.ts` |
| 患者注册 | ✅ 已实现 | `app/api/auth/register-patient/route.ts` |
| 服务端角色守卫 | ✅ 已实现 | `lib/auth/require-app-session.ts` — 4级守卫 |
| 客户端 Auth Context | ✅ 已实现 | `contexts/AuthSessionContext.tsx` — localStorage 存储 |
| 游客→注册用户升级 | ✅ 已实现 | `QuotaManager.upgradeToRegisteredUser` |
| 微信 OAuth 登录 | ❌ 未实现 | 仅在数据模型中预留 `wechat_h5` channel 标识 |
| Next.js Middleware | ❌ 未实现 | 无 middleware.ts，路由级保护缺失 |
| Token 刷新/滑动续期 | ❌ 未实现 | Token 7天过期后直接登出 |

## 二、使用场景分析

### 场景 A：家长在家自测

**环境**：家长个人手机，通过微信公众号文章链接或二维码打开 H5 页面（微信内置浏览器），或 Safari/Chrome 直接访问。

**特征**：
- 设备是个人设备，登录一次后长期保持
- 微信浏览器中 `localStorage` 基本可靠，但存在被系统清理的可能（iOS 内存压力下）
- 网络环境可能不稳定（地铁、电梯）
- 家长只应看到自己孩子的数据

**期望流程**：
1. 首次访问 → 显示登录页 → 手机号 + SMS 验证码登录
2. 7天内再次访问 → 自动续期进入首页
3. 超过7天或 Token 失效 → 回到登录页，保留上次填写的手机号
4. 如果从公众号菜单进入且已关注 → 可走微信静默授权免登录（未来）

### 场景 B：医生门诊测评

**环境**：诊所/医院的共用平板或手机，可能有多位医生轮班使用同一设备。

**特征**：
- 设备是共享设备，不能长期绑定某个账号
- 医生需要快速切换患者，中途可能被打断
- 存在"把手机递给家长填写"的交接场景
- 医生数据涉及临床信息，安全等级更高
- 网络通常为 WiFi/4G，相对稳定

**期望流程**：
1. 打开 H5 → 登录页 → 医生手机号 + SMS 验证码（或 PIN 快捷登录）
2. 进入医生工作台 → 选择患者 → 发起测评
3. 交接模式：锁定屏幕给家长 → 家长填完 → 医生 PIN 重新解锁
4. 切换医生时：退出当前账号 → 下一位医生登录
5. 长时间无操作（30分钟）→ 自动锁定，需重新验证 PIN

### 场景 C：家长到院填写（医生发起的交接模式）

**环境**：医生的设备上，家长作为"访客"临时填写量表。

**特征**：
- 家长不需要登录（是医生的操作）
- 家长只能看到当前这一份量表
- 填写完成后设备自动锁定，防止误操作
- 数据归属于医生的患者记录，而非家长账号

**期望流程**：
1. 医生选择"交接填写"模式 → 屏幕锁定为量表界面
2. 家长看到简化的填写界面（无导航、无其他数据入口）
3. 填写完成 → 显示"请将手机交还医生"页面
4. 医生输入 PIN 码解锁 → 查看报告

## 三、登录页面设计

### 3.1 新增页面清单

| 页面 | ScreenId | 用途 |
|------|----------|------|
| 登录首页 | `login` | 手机号输入 + SMS 验证码，可选密码登录 |
| 角色选择 | `role-select` | 登录后若身份不明确，选择"家长"或"医生" |
| 医生 PIN 快捷登录 | `doctor-pin-login` | 已登录医生的快速重入（替代完整 SMS 流程） |

### 3.2 登录页 UI 要求

**视觉方向**：延续现有 medical-soft 风格（鼠尾草绿 + 天蓝 + 奶油白），居中卡片布局，上方放产品 Logo 和"智伴童行"标题，下方为表单区域。

**交互细节**：
- 手机号输入框：`tel` 类型，自动聚焦，11位中国手机号校验
- 验证码输入框：6位数字，自动聚焦，60秒倒计时重发按钮
- "发送验证码"按钮：手机号合法后才可点击，点击后变灰倒计时
- 登录按钮：验证码满6位后激活
- 底部辅助链接：密码登录 | 注册新账号 | 游客快速体验
- 错误提示：手机号下方红字（号码格式错误），验证码下方红字（验证码错误/过期）
- 登录成功后：Token 存入 localStorage，跳转到角色对应的首页

### 3.3 登录流程状态机

```
[未认证] → 输入手机号 → 发送验证码 → 输入验证码 → 调用 login-sms API
   ↓                                                        ↓
   └── 手机号未注册 ← 404 ←──────────────────────────── 查找用户
          ↓
       跳转注册页（手机号已回填）
   
[已获取 Token] → 存入 localStorage → 调用 /api/auth/me 获取用户信息
   ↓
   ├── isDoctor && isPatient → 显示角色选择页
   ├── isDoctor → 进入 doctor-home
   └── isPatient → 进入 home
```

## 四、Token 管理与会话策略

### 4.1 Token 存储方案

| 存储方式 | 适用场景 | 说明 |
|----------|----------|------|
| `localStorage` | 主方案 | 微信浏览器和普通浏览器均可用，7天 TTL 与 AUS Token 对齐 |
| `sessionStorage` | 医生共享设备模式 | 医生登录时提供"共享设备"勾选项，勾选后用 sessionStorage，关闭浏览器即失效 |
| `cookie (httpOnly)` | 未来方案 | 需要后端配合设置，安全性最高但实现成本大 |

### 4.2 Token 刷新策略

当前 AUS Token 无刷新机制。建议实现"滑动续期"：
- 每次 API 调用成功时，后端返回新的 Token（剩余 TTL < 3天时触发）
- 前端在 `assessmentService.ts` 的 fetch wrapper 中统一处理：检测 response header `X-New-Token`，有则替换 localStorage 中的旧 Token
- 这样活跃用户永远不会被踢出，只有连续7天不使用才需重新登录

### 4.3 多设备/多标签页

- 不限制同时登录设备数（家长可能在手机和平板都登录）
- 使用 `storage` 事件监听跨标签页同步：一个标签页登出时，其他标签页同步登出
- 不做互踢逻辑（非金融级安全需求）

### 4.4 会话超时策略

| 角色 | 空闲超时 | 处理方式 |
|------|----------|----------|
| 家长 | 不限制 | Token 7天有效即可，无需空闲超时 |
| 医生 | 30分钟 | 显示锁屏遮罩，输入 PIN 码或重新 SMS 验证解锁 |
| 交接模式中的家长 | 不超时 | 锁定界面无法退出，直到填完交还医生 |

## 五、路由守卫与权限控制

### 5.1 新增 AuthContext

在 prototype 中创建轻量级 AuthContext（对齐主项目 `AuthSessionContext.tsx` 的接口设计）：

```typescript
interface AuthState {
  user: AuthUser | null
  token: string | null
  isAuthenticated: boolean
  isDoctor: boolean
  isPatient: boolean
  isSharedDevice: boolean    // 医生是否勾选"共享设备"
  login: (token: string) => Promise<void>
  logout: () => void
  lockScreen: () => void     // 医生空闲锁定
  unlockScreen: () => void
}
```

### 5.2 路由守卫规则

在 App.tsx 的 `renderScreen()` 最前面插入守卫逻辑：

```
if (!isAuthenticated) → 显示 LoginScreen
if (screenLocked) → 显示 LockScreen（医生 PIN 解锁）
if (isPatient && tryingToAccessDoctorScreen) → 重定向到 home
if (isDoctor && tryingToAccessPatientScreen) → 重定向到 doctor-home
```

### 5.3 移除自由切换的模式开关

当前顶部的"家长端/医生端" pill 切换必须受角色控制：
- 仅双角色用户（isDoctor && isPatient）才显示切换器
- 纯家长用户看不到医生端入口
- 纯医生用户看不到家长端入口（或可选显示只读的家长视角）

## 六、数据隔离

### 6.1 家长端数据隔离

| 数据 | 隔离规则 | Service 层实现 |
|------|----------|----------------|
| 孩子列表 | 仅返回当前用户关联的孩子 | `getChildren()` 携带 authHeaders |
| 量表列表 | 公开数据，无需隔离 | `getScales()` 不变 |
| 测评记录 | 仅返回当前用户孩子的记录 | `getHistory()` 携带 authHeaders |
| 测评报告 | 仅返回当前用户的报告 | `getReport()` 校验 childId 归属 |
| AI 解释 | 基于题目内容生成，无需隔离 | 不变 |

### 6.2 医生端数据隔离

| 数据 | 隔离规则 | Service 层实现 |
|------|----------|----------------|
| 患者列表 | 仅返回该医生关联的患者 | `getDoctorPatients()` 携带 authHeaders |
| 临时患者 | 归属当前医生的门诊会话 | `createTemporaryPatient()` 绑定 doctorSessionId |
| 测评统计 | 仅统计该医生的数据 | `getDoctorStats()` 携带 authHeaders |
| 测评历史 | 仅该医生发起的记录 | `getDoctorHistory()` 携带 authHeaders |
| 交接模式 | 医生 PIN 码保护，家长无法看到其他数据 | `verifyDoctorPin()` 本地 + 远程双重验证 |

### 6.3 交接模式特殊隔离

在 `caregiver_handoff_locked` 模式下：
- 家长看到的界面只包含当前量表的题目，所有导航、AI 助手（部分隐藏解释入口）、历史记录均不可见
- 已有的 `data-hide-in-locked` CSS 机制保持不变
- 填写过程中不暴露患者姓名或其他敏感信息
- 完成后显示交接页，等待医生 PIN 解锁

## 七、微信环境适配

### 7.1 当前可实现的（无需微信 OAuth）

- SMS 验证码登录在微信浏览器中完全可用
- 微信浏览器的 `localStorage` 基本可靠（iOS 14+ 和 Android 均支持）
- 可使用 `tel:` 协议调起拨号界面辅助输入手机号

### 7.2 微信浏览器特殊处理

| 问题 | 解决方案 |
|------|----------|
| iOS 微信 `localStorage` 在内存不足时可能被清空 | Token 同时写入 `localStorage` + `sessionStorage`，恢复时双读 |
| 微信浏览器返回按钮不触发 `popstate` | 使用微信 JS-SDK 的 `pushHistory` 技巧或引导用户点应用内返回 |
| 微信浏览器 `100vh` 包含底部工具栏 | 已使用 `100dvh`，无需额外处理 |
| 微信浏览器缓存严重 | 登录状态变更时追加时间戳 query 参数强制刷新 |

### 7.3 未来微信 OAuth 接入（Phase 2）

接入微信公众号网页授权（OAuth 2.0）后可实现：
- 已关注公众号用户静默授权 → 自动获取 openid → 查找绑定用户 → 免登录
- 未关注用户 → 显示关注引导 → 关注后自动登录
- 需在 `ScreenId` 中预留 `wechat-auth-callback` 页面处理 OAuth 回调
- 后端需新增 `app/api/auth/login-wechat/route.ts`，调用微信 `sns/oauth2/access_token` 接口

## 八、游客模式与渐进式注册

### 8.1 游客快速体验

为降低首次使用门槛，提供游客模式：
- 登录页底部"游客快速体验"按钮
- 生成 `deviceId`（`crypto.randomUUID()`），存入 `sessionStorage`
- 进入受限的家长模式：可以浏览量表和做题，但无法查看完整报告和历史记录
- 界面顶部显示横幅："注册后可保存测评记录并查看完整报告"

### 8.2 游客→注册用户升级

- 游客做完一份量表后，在报告摘要页显示"注册保存此报告"CTA
- 点击后跳转注册页，手机号已预填（如有）
- 注册成功后，调用已有的 `guestSessionId` 升级逻辑，将游客期间的测评数据迁移到正式账号
- Service 层：`registerPatient({ phone, password, guestSessionId: deviceId })`

## 九、实现优先级

### P0（首版必须）

1. **SMS 验证码登录页面** — 手机号 + 验证码表单，调用 `POST /api/auth/login-sms`
2. **AuthContext** — 用户状态管理、Token 持久化、login/logout 方法
3. **路由守卫** — 未登录时拦截所有页面，引导到登录页
4. **角色权限控制** — 根据 `isDoctor` / `isPatient` 控制可访问的页面和模式切换
5. **Service 层 authHeaders 注入** — 所有 service 函数接受并传递 `Authorization: Bearer <token>`

### P1（首版强烈建议）

6. **医生 PIN 快捷登录** — 已登录医生在同一设备上的快速重入
7. **医生空闲锁定** — 30分钟无操作显示锁屏，PIN 解锁
8. **共享设备模式** — 医生登录时勾选，Token 存 `sessionStorage`
9. **游客模式** — 受限体验 + 注册升级引导
10. **登录失败重试与错误提示** — 网络错误、验证码过期等场景

### P2（二期）

11. **微信 OAuth 静默授权** — 需后端配合
12. **Token 滑动续期** — 需后端配合返回新 Token
13. **生物识别快捷登录** — WebAuthn / 面容 ID（需 HTTPS + 支持浏览器）
